RESTful架构

URL定位资源，用HTTP动词（GET,POST,DELETE,DETC）描述操作。
RESTful架构：
　　（1）每一个URI代表一种资源；
　　（2）客户端和服务器之间，传递这种资源的某种表现层；
　　（3）客户端通过四个HTTP动词，对服务器端资源进行操作，实现”表现层状态转化”。
就是目前最流行的一个用于互联网软件的架构，它结构清晰，符合标准，易于理解。
REST全程为Respresentational State Transfer,中文含义为表现层状态转化。表现层其实指的是资源的表现层，符合REST设计规范的我们称之为RESTful设计。如果一个架构符合REST原则，就称它为RESTful架构。它的设计哲学主要是将服务端提供的内容实体看作一个资源，并表现在url（同一资源定位符）上。所谓上网就是与互联网上的一系列资源进行互动，调用它的URL，每个URI都对应一个特定的资源。要获取这个资源，访问它的URI就可以，因此URI就成了每一个资源的地址或识别符。
资源呈现出来的形式成为它的表现层。在HTTTP请求的头部信息中用Accept和Content-Type字段指定。
有无状态的理解：
1.无状态
对于每个资源的请求都不依赖于其他资源和其他请求，每个资源都是可寻址的，都有至少一个url能对其定位。
2.有状态
后面的每一个状态都依赖于前面的状态，没有一个url直接定位到资源

RESTful设计指南

1.采用https协议
2.将api部署在专有域名下，如果api简单没有后续的扩展，可以将api放在主域名下
3.将api的版本号放入url
4.路径又称终点，表示api的具体网址，在RESTful架构中，每个网址代表一种资源（resource），所以网址中不能有动词，只能有名词，而且所用的名词往往与数据库的表格名对应。一般来说，数据库中的表都是同种记录的”集合”（collection），所以API中的名词也应该使用复数。
5.对于资源的具体操作，数据的元操作，即CRUD(create, read, update和delete,即数据的增删查改)操作用HTTP动词来表示(括号是sql语句)仅通过HTTP方法，就可以完成对数据的所有增删查改工作。

  • GET（SELECT）：从服务器取出资源（一项或多项）。
  • POST（CREATE）：在服务器新建一个资源。
  • PUT（UPDATE）：在服务器更新资源（客户端提供改变后的完整资源）。
  • PATCH（UPDATE）：在服务器更新资源（客户端提供改变的属性）。
  • DELETE（DELETE）：从服务器删除资源。
  • HEAD：获取资源的元数据。
  • OPTIONS：获取信息，关于资源的哪些属性是客户端可以改变的。

6.过滤信息常见的参数

  • ?limit=10：指定返回记录的数量
  • ?offset=10：指定返回记录的开始位置。
  • ?page=2&per_page=100：指定第几页，以及每页的记录数。
  • ?sortby=name&order=asc：指定返回结果按照哪个属性排序，以及排序顺序。
  • ?animal_type_id=1：指定筛选条件

7.服务器向用户返回的状态码和提示信息，常见的有以下一些（方括号中是该状态码对应的HTTP动词）。

  • 200 OK - [GET]：服务器成功返回用户请求的数据，该操作是幂等的（Idempotent）。
  • 201 CREATED - [POST/PUT/PATCH]：用户新建或修改数据成功。
  • 202 Accepted - [*]：表示一个请求已经进入后台排队（异步任务）
  • 204 NO CONTENT - [DELETE]：用户删除数据成功。
  • 400 INVALID REQUEST - [POST/PUT/PATCH]：用户发出的请求有错误，服务器没有进行新建或修改数据的操作，该操作是幂等的。
  • 401 Unauthorized - [*]：表示用户没有权限（令牌、用户名、密码错误）。
  • 403 Forbidden - [*] 表示用户得到授权（与401错误相对），但是访问是被禁止的。
  • 404 NOT FOUND - [*]：用户发出的请求针对的是不存在的记录，服务器没有进行操作，该操作是幂等的。
  • 406 Not Acceptable - [GET]：用户请求的格式不可得（比如用户请求JSON格式，但是只有XML格式）。
  • 410 Gone -[GET]：用户请求的资源被永久删除，且不会再得到的。
  • 422 Unprocesable entity - [POST/PUT/PATCH] 当创建一个对象时，发生一个验证错误。
  • 500 INTERNAL SERVER ERROR - [*]：服务器发生错误，用户将无法判断发出的请求是否成功。

8.针对不同操作，服务器向用户返回的结果应该符合以下规范。

  • GET /collection：返回资源对象的列表（数组）
  • GET /collection/resource：返回单个资源对象
  • POST /collection：返回新生成的资源对象
  • PUT /collection/resource：返回完整的资源对象
  • PATCH /collection/resource：返回完整的资源对象
  • DELETE /collection/resource：返回一个空文档
    9.序列化和反序列化
    在服务器和客户端内部数据的格式是native类型的数据格式，将客户端和服务器端的数据传输的过程是将其转化为RESTful api标准格式的数据也就是用的最多的JOSN和XML。其中，将native类型数据转为json即为序列化，将json转为native类型数据即为反序列化。
    10.数据校验
    当客户端发送http请求时，通常也会发送一些json格式的数据，服务器在处理接受的数据之前，往往要对数据进行校验，那么校验的类型通常有类型，格式，逻辑，
    11.用户验证和权限机制

    OAuth2.0认证机制
    权限机制是对API请求更近一步的限制，全局型的权限机制，主要指通过为用户赋予权限，或者为用户赋予角色或划分到用户组，然后为角色或用户组赋予权限的方式来实现权限控制，对象型的权限机制，主要指权限控制的颗粒度在object上，用户对某个具体对象的访问、修改、删除或其行为，要单独在该对象上为用户赋予相关权限来实现权限控制。
    12.跨域解决
    CORS即Cross-origin resource sharing跨越资源共享，在RESTful API开发中，主要是为js服务的，解决javascript调用 RESTful API时的跨域问题。
    13.下面是一些例子
    例如：
    /users/jacksontian
    这这个资源的操作，主要体现在http请求的方法上。不是体现在url上。过去的实现方法：
    POST /user/add?username=jacksontian
    GET /user/remove?username=jacksontian
    POST /user/update?username=jacksontian
    GET /user/get?username=jacksontian
    而在RESTful中：
    POST /user/jacksontian
    DELETE /user/jacksontian
    PUT /user/jacksontian
    GET /user/jacksontian
    另外在资源的具体表现形态上。过去是主要表现在url的文件后缀上，设计资源的格式与后缀有很大的关联。例如：
    GET /user/jacksontian.json
    GET /user/jacksontian.xml
    而在RESTful设计中，资源的具体格式是由请求报头Accept字段和服务器端的支持情况来决定的。例如：
    Accept：application/json,application/json
    服务端根据自己能响应的格式做出响应，在响应报文中通过Content-Type字段告知客户端是什么格式。
    Content-Type：application/json;
    综上所述：REST设计就是通过URL设计资源，请求方法定义资源的操作， 通过Accept决定资源的表现形式。相MVC,RESTful只是将http请求方法加入了路由的过程，以及url路径上的更资源化。
